# What stays Zero

Both runtimes use Zero’s local reactive store, typed queries, optimistic mutations, and client synchronization model. Their server implementations differ.

## Compatibility at a glance

| Surface                         | Node                        | Orez Lite                                         |
| ------------------------------- | --------------------------- | ------------------------------------------------- |
| `@rocicorp/zero` client         | Stock                       | Stock client with Orez Lite transport integration |
| Local query and optimistic APIs | Stock                       | Preserved                                         |
| Server engine                   | Stock zero-cache package    | Orez Lite                                         |
| Change source                   | PostgreSQL replication      | SQLite change log                                 |
| Server state                    | Stock CVR and IVM internals | Orez Lite cookies and query membership            |
| Server transport                | Zero websocket/HTTP         | Pull, push, and wake                              |

## Node compatibility

The Node runtime uses the stock Zero protocol and zero-cache package. Embedded Postgres supplies native logical replication. PGlite supplies the database and replication interfaces used by the same zero-cache process.

## Orez Lite implements the client contracts

Orez Lite uses its own change log, cookies, and mutation IDs. Its query membership and permission code determine which rows enter a client patch. Compatibility is qualified through observable client behavior and named protocol scenarios for each release.

<Callout title="Versioning boundary">
  The `orez` package declares `@rocicorp/zero >=1.0` as its install-time peer range.
  Pin and qualify the exact Orez and Zero versions used by a project, because internal Zero
  changes can require an Orez release.
</Callout>
