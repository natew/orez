# SQLite sync engine

The orez sync engine is a Rust implementation of Zero's HTTP pull and push
protocol over SQLite. It leaves `@rocicorp/zero` unchanged and replaces the
server side: change tracking, incremental pulls, custom-mutator pushes,
last-mutation IDs, and query-aware synchronization.

On Cloudflare, one Durable Object owns one namespace and its SQLite database.
The engine is compiled to WASM and the TypeScript host handles authentication,
routing, service bindings, and lifecycle.

## Two write models

A deployment chooses exactly one model. The host rejects configs that combine
them or provide neither.

### Local mutators

Register mutation functions with the host. A push runs the named mutator inside
the same SQLite transaction that advances its last-mutation ID and change-log
watermark.

This is the smaller architecture: one host is the system of record, writes are
atomic, and no replication feed is needed. It works when every authoritative
write can run inside the Durable Object.

The cost is ownership. Background jobs, server actions, and other writers must
also enter through this host or they will write somewhere clients never read.

### Delegated writes

Set `mutateUrl` and `upstream`. The sync host forwards pushes to the application's
authoritative mutation endpoint, then ingests committed rows from its change
feed. Authentication, permission transforms, and business logic remain with
the application.

This mirrors the split used by zero-cache and is the right model when several
systems already write to one upstream database. It adds replication lag, feed
retention, and snapshot recovery as operational concerns.

## Pulls and cookies

Each namespace has a monotonic change-log watermark. A pull can return:

- `unchanged` when the client's cookie equals the watermark;
- a primary-key diff when the cookie is inside the retained window;
- a full snapshot for a new client, a cookie below the retained floor, or an
  invalidated visibility epoch;
- HTTP 409 when a cookie is ahead of this engine's watermark.

The log records touched primary keys, not row images. Pulls read live rows in
the same SQLite transaction, avoiding stale values and JSON float round-trips.

## Query awareness and visibility

`resolveQuery` can turn a named client query and arguments into a validated Zero
AST after authentication. A server-owned `queryTransformVersion` invalidates
previous transforms when permissions or schema rules change.

For simpler row-local permissions, `visibility.filter` returns a SQL predicate
per table and user. Because permission changes can revoke a row without changing
that row, visibility-filtered pulls use snapshots rather than diffs.

## Recovery

Delegated ingest stores its upstream cursor separately from the client-facing
cookie. When a cursor falls behind the source feed's retained floor, the host
requests a full snapshot, replaces application tables atomically, and resumes
the change stream from the snapshot watermark. Clients never see the upstream
cursor domain.

## Deployment fit

The engine fits workloads that partition naturally into namespaces, fit within
one Durable Object per namespace, and have moderate per-namespace writes. It is
not a transparent replacement for a single high-throughput database. Review
[Trade-offs](/docs/trade-offs), especially single-writer throughput and
Cloudflare billable-row amplification, before choosing it.
