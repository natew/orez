# Cloudflare setup

<Status tone="preview">Server host · Preview</Status>

The Cloudflare host runs Orez Lite's Rust/WASM engine inside a SQLite Durable
Object. It replaces the `zero-cache` deployment. The browser still runs the
real `@rocicorp/zero` client, and the application still exposes its ordinary
Zero query and mutate endpoints.

<Callout title="Zero stays the application contract" tone="note">
  Follow Zero's [installation guide](https://zero.rocicorp.dev/docs/install) for the
  schema, client, queries, mutators, and application endpoints. This page covers only the
  Orez Lite server boundary.
</Callout>

## Server boundary

The host package exports `createSyncWorker` and `createSyncDurableObject`.
Application-specific server configuration supplies:

- the Zero schema and replica DDL;
- authentication and namespace resolution;
- authorization for wake sockets, upstream notifications, and admin routes;
- the existing application query and mutate endpoints;
- a change feed from the authoritative database.

```text
real Zero client ── Zero protocol ──→ Orez Lite
                                         │
                         query and mutate│requests
                                         ▼
                                  application server
                                         │
                                  authoritative writes
                                         ▼
                              database change feed ──→ Orez Lite replica
```

Orez forwards authenticated query and mutation work to the application rather
than copying application logic into the sync host. Committed rows return through
the database change feed.

## Client connection

Set the existing Zero client's `cacheURL` to the deployed Orez Lite address.
There is no Orez client package or alternate query API. Components keep using
the same `@rocicorp/zero` queries, mutators, hooks, and optimistic local state.

## Namespaces and authorization

A namespace is a storage and ordering boundary, commonly an organization or
project. It is not an authorization mechanism. The outer Worker must
authenticate requests before selecting a Durable Object, and the application
must still authorize every query and mutation.

Wake sockets and upstream notifications are fail-closed. Consumers must provide
`authorizeWake` and `authorizeNotify`; rejected requests do not instantiate a
namespace Durable Object. See the
[`orez-sync-cf-host` integration guide](https://github.com/natew/orez/tree/main/packages/sync-cf-host)
for the capability and service-binding patterns.

## Deployment status

The public integration uses a service binding for application queries and
mutations plus a change feed from the authoritative data service. Packaging the
consumer adapter and stable configuration surface is still in progress, so the
host remains preview.

Application code should not depend on Orez's internal query AST, replication
cursor, Durable Object schema, or cache invalidation tables. Those are server
implementation details.
