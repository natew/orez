# Orez Lite setup

<Status tone="preview">Cloudflare host · Preview</Status>

Start with a working Zero or `on-zero` application. Orez Lite replaces the
`zero-cache` synchronization engine. It does not replace your schema, named ZQL
queries, custom mutators, authentication, or application endpoints.

<Callout title="Before you start" tone="note">
  If the application does not yet have a Zero query endpoint and mutate endpoint, follow
  Zero’s [installation guide](https://zero.rocicorp.dev/docs/install) first.
</Callout>

## Application surface

The application keeps these pieces:

- its Zero schema and relationships;
- registered named queries that return ZQL;
- registered custom mutators;
- an authenticated query endpoint, commonly `/api/zero/pull` in `on-zero` apps;
- an authenticated mutate endpoint, commonly `/api/zero/push`;
- one authoritative database.

Permissions stay in the application. Read permissions are filters in the ZQL
resolved by the query endpoint. Write permissions are checks in the server-side
mutator. See [Queries and permissions](/docs/orez-lite/queries) and
[Mutations](/docs/orez-lite/writes).

## Client connection

The client still uses the stock `@rocicorp/zero` client and the same registered
queries and mutators. `on-zero` applications install its Orez HTTP transport for
the deployed namespace, then point the provider at that URL:

```tsx
import { ensureHttpPullTransport, ProvideZero } from 'on-zero'

export function OrezRoot({ children, namespace, user, sessionToken }) {
  const cacheURL = `${OREZ_SYNC_URL}/${namespace}`

  if (typeof window !== 'undefined') {
    ensureHttpPullTransport({
      origin: cacheURL,
      queryForward: true,
      wake: true,
    })
  }

  return (
    <ProvideZero
      cacheURL={cacheURL}
      transport="http-pull"
      userID={user.id}
      auth={sessionToken}
      authData={user}
    >
      {children}
    </ProvideZero>
  )
}
```

`queryForward` sends registered query names and arguments to the application
query endpoint. `wake` lets the host notify the client to pull immediately after
new data arrives. Application components continue to call `useQuery(...)` and
`zero.mutate(...)` exactly as before.

## Deployment connection

The Cloudflare deployment connects the sync host to the application over
private service bindings:

```text
                         ┌─ query name + args ─→ app query endpoint
Zero client ─→ Orez Lite ┤
                         └─ mutation push ─────→ app mutate endpoint
                                                      │
                                                      ▼
                                             authoritative database
                                                      │
                                                      ▼ changes
                         client patches ←──── Orez Lite replica
```

The application endpoint authenticates the original bearer token. Orez receives
only the authorized query result shape and committed database changes needed to
maintain client subscriptions.

The deployment also chooses a namespace key, such as an organization or
project ID. A namespace is a storage and ordering boundary, not an authorization
mechanism. The application must still authenticate every query and mutation.

## What is currently Cloudflare-specific

The public Orez Lite integration runs the Rust/WASM engine in a SQLite Durable
Object. It uses a service binding for application queries and pushes, plus a
change feed from the authoritative data service. Packaging this deployment into
a smaller reusable adapter is still in progress, so the host remains marked
preview.

Application code should not depend on the host’s internal query AST, replication
cursor, Durable Object schema, or cache-invalidation controls. Those are Orez
implementation details and may change during preview.
