# Native setup

The native host is a Rust library around `sync-core`, Axum, and bundled `rusqlite`. It can bind its own loopback server or return a router for nesting inside an existing application.

<Callout title="Source integration today">
  The native crates currently live in the Orez workspace as `sync-core` and `sync-native`.
  Treat the API as preview while packaging, naming, and host parity settle.
</Callout>

## Define the schema

The engine needs each synchronized table’s ordered columns, Zero value types, and primary key:

```rust
use sync_core::schema::{TableSpec, Tables};
use sync_core::value::ZeroColumnType;

let tables = Tables::new().with(
    "todo",
    TableSpec {
        columns: vec![
            ("id".into(), ZeroColumnType::String),
            ("title".into(), ZeroColumnType::String),
            ("complete".into(), ZeroColumnType::Bool),
        ],
        primary_key: vec!["id".into()],
    },
);
```

`Tables::from_zero_schema()` can consume a serialized Zero `createSchema()` result. Identifiers must be ordinary SQL identifiers. The engine rejects malformed names before creating trigger DDL.

## Configure the host

`SyncNativeConfig` supplies the application-specific seams:

```rust
use std::{path::PathBuf, sync::Arc};
use sync_native::{AuthFn, SyncNativeConfig, SyncNativeHost};

let config = SyncNativeConfig {
    tables,
    initialize: Arc::new(|db| {
        db.exec(
            "CREATE TABLE IF NOT EXISTS todo (\
               id TEXT PRIMARY KEY, title TEXT NOT NULL, complete INTEGER NOT NULL\
             )",
            &[],
        )
        .map_err(|error| error.0)?;
        Ok(())
    }),
    mutate: Arc::new(|db, name, args, user_id| {
        // Validate the named mutation, then use parameterized db.exec/db.query.
        run_app_mutator(db, name, args, user_id)
    }),
    visible: None,
    authenticate: Arc::new(|headers| {
        headers
            .get("authorization")?
            .to_str()
            .ok()?
            .strip_prefix("Bearer ")
            .map(str::to_owned)
    }) as AuthFn,
    retain_changes: 4096,
    visibility_enabled: false,
    query_aware: false,
};

let host = SyncNativeHost::new(config, PathBuf::from(".orez-lite"));
host.run(8788).await;
```

The example’s `run_app_mutator` is application code. Return `MutateError::app(message)` for a rejected application mutation. Its mutation ID still advances. Return `MutateError::Other(message)` for infrastructure failure. The entire push then rolls back and can retry.

## Embed the router

To share a process and listener with another Axum service:

```rust
let sync = SyncNativeHost::new(config, data_dir).into_router();

let app = axum::Router::new()
    .nest("/sync", sync)
    .route("/health", axum::routing::get(app_health));
```

The public protocol routes are `/{namespace}/pull`, `/{namespace}/push`, and `/{namespace}/wake`. The host also exposes health, status, invalidation, and fault-oriented admin routes. Protect operator endpoints with an application authorization boundary.

## Storage and concurrency

Each namespace maps to `<data-dir>/<namespace>.sqlite` in WAL mode. A namespace owns one serialized writer worker, while separate namespaces can progress independently. This keeps transaction order clear and protects the connection from concurrent use. The single writer sets the throughput ceiling for a hot namespace.

## Native scope today

The native host supports local mutators that execute inside the SQLite transaction. Each push applies application data, change tracking, and mutation IDs atomically. See [Writes and mutators](/docs/orez-lite/writes) for the local authority model.
