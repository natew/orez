# Architecture

Orez Lite changes the synchronization machinery while leaving application
queries, mutations, authentication, and permissions in the application.

## Request and data flow

```text
                           query name + args
                      ┌────────────────────────→ app query endpoint
                      │                              │ authorized ZQL
Zero client ──HTTP──→ Orez Lite sync host ←──────────┘
   ▲                  │
   │ row patches      │ mutation push
   │                  └────────────────────────→ app mutate endpoint
   │                                                 │ commit
   │                                                 ▼
   └──────────── query tracking ← change feed ← authoritative database
```

The two application endpoints are the same endpoints used with stock
`zero-cache`. Orez forwards the original authenticated request instead of
reimplementing application logic.

## Query path

The client sends a registered query name and arguments. The application query
endpoint authenticates the request and resolves that name to authorized ZQL.
Orez tracks the resulting row membership and sends an initial snapshot followed
by incremental patches.

The engine stores an internal query representation and reference counts, but
application authors work only with named queries and ZQL. See
[Queries and permissions](/docs/orez-lite/queries).

## Mutation path

The client runs its registered mutator optimistically, then Orez forwards the
mutation to the application push endpoint. The endpoint authenticates the
request and executes the server-side mutator in the authoritative database.
Committed rows return through the database change feed.

This preserves one mutation implementation and one write authority. See
[Mutations](/docs/orez-lite/writes).

## SQLite replica

The Cloudflare host keeps a synchronized SQLite copy per namespace. Database
changes enter the replica in commit order and append touched table/primary-key
entries to the engine change log. Pull reads the current row in the same
transaction, so intermediate values do not leak into client patches.

## Cookies and retention

Each namespace has a monotonic watermark and a retained floor:

- cookie equals watermark: no changes;
- cookie is inside retained history: incremental touched-key diff;
- new or expired cookie: full snapshot;
- cookie is ahead of the server: reset response.

Pruning removes old change rows while preserving the durable high-water mark.

## Mutation acknowledgements

The engine tracks last-mutation IDs by client group and client. It records the
application push result independently from row arrival through the change feed.
This makes retries idempotent while keeping the application database as the only
row authority.

## Wake channel

Wake is a data-available hint. The client follows it with a pull. Periodic safety
pulls cover delayed, coalesced, duplicated, or missed wake frames; state and
cookies always travel through pull responses.
