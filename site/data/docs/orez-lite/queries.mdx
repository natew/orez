# Queries and permissions

Orez Lite supports baseline namespace synchronization and query-aware synchronization. Both strategies enforce permissions before rows enter a client patch.

## Baseline pulls

Baseline mode synchronizes the visible rows of configured tables. With no visibility callback, every authenticated user can see every synchronized row in the namespace.

A native visibility callback returns a SQL predicate and parameters per table and user. The Cloudflare host exposes the equivalent `visibility.filter(table, claims)` configuration.

```ts
visibility: {
  rowLocal: true,
  filter(table, claims) {
    if (table === 'todo') {
      return { sql: 'ownerId = ?', params: [claims.userID] }
    }
  },
}
```

Because a permission change can revoke an untouched row, visibility-filtered baseline pulls use a full snapshot to recompute the complete visible set.

## Query-aware pulls

Query-aware mode consumes desired queries from the client and compiles each allowed query into a validated Zero AST. The engine maintains membership and reference counts so the patch can add rows entering a result and remove rows leaving it.

The Cloudflare host can resolve named queries after authentication:

```ts
queryAware: true,
resolveQuery(name, args, claims) {
  if (name === 'myTodos') {
    return buildTodoAst({ ownerId: claims.userID, limit: args[0] })
  }
  throw new Error('query not allowed')
},
queryTransformVersion: 3,
```

The application validates query names and arguments, then constructs the authorized AST on the server.

## Transform invalidation

`queryTransformVersion` is a server-owned epoch. Increase it when schema or permission logic changes in a way that invalidates cached query transforms. Orez Lite then rebuilds affected membership under the current rules.

For baseline visibility, the admin invalidation route advances an engine marker so subsequent pulls rebuild from a safe snapshot.

## Namespace isolation

Namespace resolution happens before the engine opens storage. Use namespaces as a genuine ownership and scaling boundary, such as organization, workspace, or project. Authentication still applies within every namespace.

## Permission checklist

- Normalize and validate identity at the host boundary.
- Derive predicates and query ASTs on the server.
- Parameterize SQL values; never interpolate claims into SQL strings.
- Version permission transforms.
- Test both initial visibility and revocation.
- Assert that forbidden rows never appear in snapshots, diffs, logs, or operator responses.
