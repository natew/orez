# Queries and permissions

Orez Lite uses Zero’s named-query model. Define queries with ZQL, register them
with Zero or `on-zero`, and resolve them through the application query endpoint
you already use.

Read Zero’s [Queries](https://zero.rocicorp.dev/docs/queries) guide for the full
query lifecycle and the [ZQL reference](https://zero.rocicorp.dev/docs/zql) for
filters, relationships, ordering, limits, and pagination.

## Define named ZQL queries

In an `on-zero` application, exported query functions are registered by the
generated query registry. Keep read authorization in the ZQL returned by those
functions:

```ts
import { serverWhere, zql } from 'on-zero'

const canReadProject = serverWhere('project', (q, auth) =>
  q.or(
    q.cmp('ownerId', auth?.id ?? ''),
    q.exists('members', (member) => member.where('userId', auth?.id ?? ''))
  )
)

export const projectById = (args: { id: string }) =>
  zql.project.where(canReadProject).where('id', args.id).one()
```

The client sends the registered query name and validated arguments. The
application resolves that name using authenticated server context and returns
the authorized ZQL query. Client arguments are useful filters, but they are not
proof of identity or access.

## Keep the application query endpoint

The application endpoint authenticates the request before `on-zero` resolves
the registered query:

```ts
export async function POST(request: Request) {
  const authData = await getAuthData(request)
  if (!authData) return new Response('unauthorized', { status: 401 })

  const { response } = await zeroServer.handleQueryRequest({
    authData,
    request,
  })

  return Response.json(response)
}
```

With stock `zero-cache`, this is the endpoint configured by `ZERO_QUERY_URL`.
With Orez Lite, it remains the same endpoint. Orez asks the application to
resolve each named query, then maintains the resulting rows as the authoritative
database changes.

```text
useQuery(projectById, { id })
  → registered query name + arguments
  → application /api/zero/pull
  → authentication + authorized ZQL
  → Orez Lite tracks the result
  → incremental row patches reach the client
```

## Permissions are application logic

Zero does not require a separate RLS-style permission language. Read access is
expressed by the authenticated ZQL returned from the query endpoint. Write
access is enforced by the server-side mutator. See Zero’s
[permission patterns](https://zero.rocicorp.dev/docs/auth#permission-patterns)
for ownership, membership, sharing, and role examples.

Orez’s internal query representation and cache invalidation are deployment
details. Application authors should not construct query AST objects, SQL
visibility callbacks, or Orez-specific permission rules.

## Checklist

- Register every synchronized query by name.
- Validate query arguments at the server boundary.
- Derive identity and roles from the authenticated request.
- Apply read filters in server-resolved ZQL.
- Enforce write rules again in the server-side mutator.
- Test both initial access and revocation with real clients.
