# Mutations

Orez Lite does not introduce another mutation API. Define mutations with Zero or
`on-zero`, run them optimistically on the client, and execute them authoritatively
through your application’s existing push endpoint.

If you are new to this model, start with Zero’s
[Mutators](https://zero.rocicorp.dev/docs/mutators) guide. It covers validation,
registries, optimistic execution, server execution, and the push protocol.

## Define mutations in the application

An `on-zero` application can keep its existing mutation modules. This example
uses the authenticated user as the owner instead of trusting an owner ID from
the client:

```ts
import { ensureLoggedIn, mutations, serverWhere } from 'on-zero'

const canWriteTodo = serverWhere('todo', (q, auth) => q.cmp('ownerId', auth?.id ?? ''))

export const mutate = mutations('todo', canWriteTodo, {
  insert: async ({ tx }, args: { id: string; title: string }) => {
    const auth = ensureLoggedIn()
    await tx.mutate.todo.insert({
      id: args.id,
      title: args.title,
      ownerId: auth.id,
    })
  },

  update: async ({ tx }, args: { id: string; title?: string }) => {
    await tx.mutate.todo.update(args)
  },
})
```

The same definition supplies the optimistic client behavior and the
authoritative server behavior. Mutation arguments are untrusted. Authentication
and write authorization belong in the server execution of the mutator.

## Keep the application push endpoint

The application endpoint authenticates the request and lets `on-zero` execute
the named mutation against the authoritative database:

```ts
export async function POST(request: Request) {
  const authData = await getAuthData(request)
  if (!authData) return new Response('unauthorized', { status: 401 })

  const { response } = await zeroServer.handleMutationRequest({
    authData,
    request,
  })

  return Response.json(response)
}
```

With stock `zero-cache`, this is the endpoint configured by `ZERO_MUTATE_URL`.
With Orez Lite, it remains the same endpoint. The Orez deployment forwards the
push to it and replicates the committed database changes into the sync engine.

```text
zero.mutate(...)
  → optimistic local mutation
  → application /api/zero/push
  → authenticated server mutator
  → authoritative database commit
  → Orez Lite replication
  → subscribed clients update
```

The application has one write authority: its mutators and push endpoint. You do
not copy those mutators into an Orez registry or define a second permission
system in the sync host.

## Authorization

Use authenticated server context for facts the client must not choose, such as
the current user, organization, or role. Enforce ownership and other write rules
inside the mutator before changing rows. Zero’s
[Authentication](https://zero.rocicorp.dev/docs/auth) guide has the canonical
read and write authorization patterns.

## What Orez Lite owns

Orez Lite owns sync bookkeeping after the application commits: mutation IDs,
replication cursors, query updates, client patches, and wake notifications. It
does not own application validation, business rules, authorization, or side
effects.
