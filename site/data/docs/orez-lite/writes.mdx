# Writes and mutators

Orez Lite supports two authority models. Every deployment needs one clear writer path. Local application writes and a separate untracked authority cannot safely write to the same namespace.

## Local mutators

Local mutators run in the same SQLite transaction as application changes, the mutation result, the client’s last-mutation ID, and change-log entries.

```text
validate push
  → check mutation ID / replay
  → run named application mutator
  → record touched primary keys
  → advance LMID and watermark
  → commit
  → acknowledge and wake peers
```

Use this model when Orez Lite’s SQLite database is authoritative and every writer can enter through registered mutators.

An application rejection still advances the mutation ID, matching the retry behavior expected by Zero clients. An infrastructure error rolls back the whole push so it can safely retry.

## Delegated writes

The Cloudflare host can forward pushes to an authoritative application service and ingest its committed change feed:

```text
client push → Orez Lite → application mutation service
                              ↓
                         authoritative DB
                              ↓
                      retained change feed
                              ↓
                         Orez Lite SQLite
```

Configure `mutateUrl`, a service `mutateBinding`, and `upstream`. The host retries eligible delegated failures with bounded exponential backoff, then applies feed pages in order.

This model fits systems where server actions, jobs, or other services already write to an upstream database. It adds replication lag, cursor retention, snapshot recovery, and two operational systems.

<Callout title="Host difference" tone="warning">
  Delegated push and upstream ingest are currently Cloudflare-host capabilities. Native
  deployments currently use transactional local mutators.
</Callout>

## No dual apply

The Cloudflare config rejects a local `mutators` registry combined with delegated `upstream` writes. Running the same mutation in both systems can double-apply effects or let mutation acknowledgements get ahead of authoritative data.

## Side effects

Cloudflare mutators can register post-commit effects with `context.defer()`. Those effects run after the SQLite transaction commits, so they need idempotency and their own observability. A later email, queue, or webhook failure cannot alter the committed mutation.

## Picking a model

| Question                         | Local mutators                  | Delegated writes                                   |
| -------------------------------- | ------------------------------- | -------------------------------------------------- |
| Where is truth?                  | Orez Lite SQLite                | Existing upstream database                         |
| Atomic application write + LMID? | Yes, one transaction            | Split across service response and feed convergence |
| Other writers allowed?           | Only through the Orez Lite host | Yes, if the feed captures them                     |
| Operational complexity           | Lower                           | Higher                                             |
| Native host today                | Yes                             | Not yet                                            |
| Cloudflare host today            | Yes                             | Yes                                                |
