# Documentation

orez is a set of small runtimes for building with Zero without starting with a
conventional database stack. The default command runs PGlite, a PostgreSQL wire
proxy, zero-cache, and its SQLite replica. More demanding workloads can opt into
real embedded Postgres, while the separate Rust sync engine can replace
zero-cache with SQLite on Cloudflare.

```sh
bunx orez
```

## Pick a path

- **New local project:** follow [Getting started](/docs/getting-started). The
  defaults are intentionally useful without configuration.
- **Choosing storage or performance characteristics:** read [Runtime modes](/docs/modes).
- **Using orez from a config file, API, or Vite:** see [Configuration](/docs/configuration).
- **Running sync on Durable Objects:** start with the [Sync engine](/docs/sync-engine).
- **Translating PostgreSQL SQL for SQLite:** use the standalone
  [pg-to-sqlite](/docs/pg-to-sqlite) package.

## What stays compatible

Your application still uses `@rocicorp/zero`. orez changes the server-side
machinery: where Postgres runs, how zero-cache gets SQLite, or—when using the
Rust engine—which server answers the Zero pull and push protocol. The client
query API, optimistic mutations, and local store remain Zero.

## Scope

The default PGlite stack is a development runtime. It prioritizes a portable,
zero-install environment over multi-connection throughput. The embedded
Postgres mode exercises the real PostgreSQL logical-replication path. The
Cloudflare sync engine is a separate production-oriented runtime with its own
storage and workload constraints. The [modes page](/docs/modes) makes those
boundaries explicit.
