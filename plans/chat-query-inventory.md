## 1. Schema

Chat's Zero schema is generated by drizzle-zero into
`src/data/generated/schema.ts` (6070 lines; `export const schema` at `:5681`,
`export const zql = createBuilder(schema)` at `:6058`). It is **not** hand
written; the drizzle source is `~/chat/drizzle-zero.config.ts` +
`src/database/schema-public`.

### 1.1 Column types (the whole type universe)

Only four Zero column types appear across all 51 tables
(`grep "type: '...'" schema.ts`):

| Zero type | count | SQLite mapping (for M4b `toZeroValue`) |
| --- | --- | --- |
| `string` | 329 | TEXT |
| `number` | 127 | INTEGER or REAL (JS number; i64/2^53 boundary per M0) |
| `boolean` | 40 | INTEGER 0/1 |
| `json` | 38 | TEXT (parsed on read) |

No blob, no date/timestamp column type: timestamps are `number` (epoch ms,
e.g. `notification.createdAt` `schema.ts:2196-2200`). JSON columns carry
app-specific shapes via `customType` (e.g. `data.extra`, `channel.state`,
`flow.steps`); the sync engine treats them as opaque TEXT and never compares
their subfields (no query does).

### 1.3 Relationships (including junctions)

`schema.ts` declares 310 relationship edges (160 `cardinality: 'one'`, 150
`'many'`). **12 are two-hop junction (many-to-many) relationships** — a
relationship whose value is an array of two connection objects, which the
compiler must expand into the intermediate table (two joins), not one:

| Junction (source → junction table → dest) | schema.ts | Used in |
| --- | --- | --- |
| channel.`roles` → channelPermission → role | :3686 | `channelWithRoles`, `currentServer.roles` |
| channel.`channelUserRoles` → channelPermission → userRole | :3714 | **channel read permission** `where/channel.ts:43,66,89` |
| server.`members` → serverMember → userPublic | :4745 | membership listings |
| server.`installedApps` → appInstall → app | :4986 | app UI |
| userPublic.`servers` → serverMember → server | :5247 | |
| userPublic.`friends` → friendship → userPublic (self-join) | :5627 | `userWithFriends` |
| userPublic.`userRoles` → userRole → role | :5641 | roles |
| (+ role.`role`, channel.`messages`, server.`member`, userPublic.`server`, server.`servers` two-hop entries) | | |

The junction `channelUserRoles` is security-relevant: the channel read
predicate resolves "does this user have a role on this private channel" by an
`EXISTS` over it (`where/channel.ts:43`). The compiler comment at
`queryMessageItemRelations.ts:36-38` records a live Zero limitation —
"Order by is not supported in junction relationships yet" — so junction
`related()` never carries `orderBy`; the compiler does not need ordered
junction output.

### 1.4 Namespace split: control plane vs per-server namespaces

Chat runs **two Zero client instances** that sync to **different namespaces**
(the "#15 per-server tenancy" work). Wiring: `src/zero/client.tsx:25-138`,
split registry `src/zero/core.ts:22-222`.

- **Control / singleton namespace** — the global graph (identity, social,
  settings, catalog, server rail). Client mounts OUTER
  (`client.tsx:95-106`); `cacheURL = ZERO_SERVER_URL` (`client.tsx:97`).
- **Per-server namespaces** — one per server, routed on Cloudflare by the
  websocket path `/p-<namespaceId>` → DO namespace `proj-<namespaceId>`
  (`client.tsx:125-126`, `serverNamespaceId.ts`). Client mounts INNER, keyed
  by active serverId; disabled (no socket) off a server page
  (`client.tsx:116-138`).

`combineZeroClients` (`core.ts:202`) routes each query/mutator to its owning
instance **by function reference**, so the `server.*` query namespace is
split by function, not by namespace string (`core.ts:17-21`).

**Control-plane models** (`core.ts:99-114`): admin, app, appSource, device,
friendship, job, privateBetaInvite, privateChatsStats, reaction, server,
serverMember, userPublic, userState, waitlist.

**Per-server models** (`core.ts:116-157`): agentConfig, apiKey, appInstall,
appInstallPermission, approval, attachment, channel, channelActive,
channelAgentConfig, channelNotificationSetting, channelNotificationState,
channelPermission, channelTopic, channelTopicMessage, contentVersion, data,
devtools, flow, flowRun, flowRunDetail, indicator, invite, meeting,
meetingUser, message, messageChannelMention, messageReaction,
messageReactionStats, messageRoleMention, messageUserMention, notification,
pin, role, secret, seen, serverAction, serverLog, skill, thread, userRole.

A module-eval guard (`core.ts:163-184`) throws if any generated model or
query function is missing from the split, so the partition is exhaustive by
construction.

**Cross-namespace projection (critical for M4b/M4c).** Per-server queries
JOIN control-tier identity/membership (message authors, member lists,
`currentServer` roles/apps), but a per-server client only sees its own
namespace. So Chat **copies** control-tier rows INTO each server's namespace:
`src/zero/projectToServerNamespace.server.ts`. Projected tables:
`userPublic` + `userState` (`:73-86`), `app` + `appSource` (`:94-107`),
`server` + `serverMember` (+ member identities) (`:127-144`). Consequence for
the Rust engine: **within a single per-server namespace the SQLite database
already contains the projected copies**, so the cross-table permission
`EXISTS` over `serverMembers` / `userRoles` / `channelUserRoles` and all
`related()` joins resolve locally. The compiler never needs to reach across
namespaces; it needs those projected tables to exist in the namespace it runs
against. (Secrets are the exception: `data.permissions='secret'` rows are
never synced — `where/data.ts:27-29`.)

## 2. Query registration (synced / named queries)

`on-zero generate` emits two files (both auto-generated; do not edit):

- `src/data/generated/groupedQueries.ts` — re-exports each query module under
  a stable namespace (`export * as channel from '../queries/channel'`), giving
  minification-safe identity.
- `src/data/generated/syncedQueries.ts` — the registration. For every table
  namespace it builds `defineQuery(validator, resolver)` entries and bundles
  them with `defineQueries({...})` exported as `queries` (`:975-1013`).

Each named query = `defineQuery(v.object({...valibot arg schema...}),
({args}) => Queries.<module>.<fn>(args))`. Example `channel.channelById`:
`syncedQueries.ts:118-123` maps name `channelById` → arg validator
`{channelId: string}` → `Queries.channel.channelById(args)`. Zero-arg queries
use `defineQuery(() => ...)` (`:421`).

The **arg validators in `syncedQueries.ts` are the authoritative parameter
shapes** for every query and are the exact bindings M4b must accept (all
scalars, arrays of scalars, and small literal unions; no nested objects reach
the query builder as bind params). No `syncedQuery`/`namedQuery`/`createQueries`
identifiers exist — the wrapper is `defineQuery`/`defineQueries` from on-zero.

Client subscription API: `useQuery` / `getQuery` / `usePermission` / `run` are
destructured from `combineZeroClients(...)` at `core.ts:214`, re-exported from
`~/zero/client` (`client.tsx:33-41`). Clients pass the **plain query function**
to `useQuery`; the server registers the validated synced query for
pull/permission enforcement. **142 `useQuery` call sites** exist across
`src`+`app`; 138 use a named query, 3 are the virtual-list generic hooks
(`src/zero/virtual-list/useRows.ts:73,93,100`), 1 is a runtime-dynamic sandbox
query (`src/features/markdown/MarkdownSandboxQuery.tsx:86`, see §3.4).

## 3. Query shapes and AST features

37 query files, ~120 exported query functions. Legend: **W** `.where`,
**WE** `.whereExists`, **EX** `.exists` (inside a predicate), **R**
`.related`, **O** `.orderBy`, **L** `.limit`, **ONE** `.one`, **cursor** =
manual `.where(col, op, val)` pagination. `.start()` is used **nowhere**.

### 3.1 Operator usage across all query files (empirical)

Grep of `src/data/queries/`:

| Operator | count | files | example |
| --- | --- | --- | --- |
| `=` (implicit + explicit) | pervasive | all | `.where('id', props.channelId)` `channel.ts:26` |
| `!=` | pervasive | many | `.where('deleted', '!=', true)` `channel.ts:34` |
| `<` `<=` `>` `>=` | 15 | data, channel, server, message, channelTopic | `.where('createdAt', '>=', x)` `data.ts:54` |
| `IN` | 7 | channel, data, message, app | `.where('id', 'IN', props.channelIds)` `channel.ts:39` |
| `LIKE` | 3 | data, devtools, user | `.where('key', 'LIKE', props.keyPattern)` `data.ts:181` |
| `ILIKE` | 11 | data, channel, message, server, user | `.where('username', 'ILIKE', '%'+s+'%')` `server.ts:90` |
| `IS` (NULL) | (of 9) | server, message | `.where('userId', 'IS', null)` `server.ts:143` |

### 3.2 The most AST-demanding queries (representative, not exhaustive)

- **`channel.channelMessages`** (`queries/channel.ts:175-235`) — the hardest.
  `channel` W(readPermission) W(id) ONE, then R(`messages`) with conditional
  W on `isThreadReply`/`type`(`!=`)/`order`(`>=`,`<`)/`threadId`(`IS NOT`)/
  `isPinned`/`creatorId`, an inline `.or`, O(`order` asc|desc), L(limit|200),
  then `queryMessageItemRelations` which adds ~11 more nested relations
  (`queryMessageItemRelations.ts:8-53`): flows→run, pin, thread→appInstalls→app,
  reactionStats→reaction, creator→roles (**junction**), replyingTo→creator,
  attachments, appInstall→app, flowRun→flow. Manual `order` cursor
  (before/after), no `.start()`.
- **`server.currentServer`** (`queries/server.ts:35-43`) — readPermission is a
  deep boolean: `_.or(_.and(cmp,_.or(cmp,cmp)), _.and(_.not(cmp),cmp),
  _.and(_.not(cmp),cmp,_.exists('member',...)))` (`server.ts:18-33`) — NOT +
  correlated EXISTS + nested and/or. Then R(member.one, roles.limit(100),
  apps→install.one).
- **`channel.channelMembers`** (`queries/channel.ts:82-131`) — 3-level
  correlated EXISTS `user → roles → channelPermissions`
  (`channel.ts:104-108`), ILIKE search on two columns under an `.or`, manual
  `joinedAt` cursor, O+L, R(user.one).
- **`data.dataSearch`** (`queries/data.ts:64-110`) — dynamic `sortColumn`
  (runtime string), dynamic cursor direction, ~10 optional filters incl.
  `ILIKE` and range `>=`/`<=` (`data.ts:23-60`), `.min(1000, limit)` cap.
- **`message.hasMessageReadPermission`** path — solo-channel check uses
  `_.not(_.exists('channel', c => c.where('solo', true)))` OR creator OR
  bot (`where/message.ts:10-16`): NOT EXISTS combined with equality.

### 3.3 Full query list

The two read-only enumeration passes produced a per-function AST matrix for
all ~120 queries; rather than restate it here, the authoritative source is
`src/data/queries/*.ts` (the arg shapes are in `syncedQueries.ts`). Coverage

- **`limit`**: `channelsByIds` L(500), `serverMembers` L(limit??100),

- **`orderBy`**: static (`createdAt`, `order`, `joinedAt`, `score`,

- **`IN`**: `channelsByIds`, `messagesByIds`, `data.dataByKeys`,

- **`LIKE`/`ILIKE`**: `data.dataByKeyLike` (LIKE), `data.dataSearch`,

- **`IS`/`IS NOT` null**: `server.searchJoinableServers`/`userServers`/

- **correlated EXISTS / whereExists**: every `where/*` predicate (§4) plus
  `channelMembers`, `serverAgentMembers`, `personalServerByUsername`,

- **related (incl. junction)**: 93 sites; junction `related` in

- **manual cursor pagination**: `channelMembers`, `serverMembers`,
  `data.dataSearch`, `channel.channelMessages` (order-based),

### 3.4 Queries outside `src/data/queries` (dynamic / inline)

  suite. M4b should treat this as out of the supported surface (reject or scope
  out); it is not production query traffic.

## 4. Permission predicates (`src/data/where/`)

These compile (via `serverWhere` → on-zero `where`, `serverWhere.ts:16-21`)
into a Zero `Condition` AST that the consumer host merges into the query's
`where` before handing the transformed AST to Rust. They are the security
core: **most are cross-table `EXISTS`, not row-local**, which is exactly why
Chat needs query-aware sync (uniform per-namespace visibility does not hold).

The `_` builder methods used: `.exists(rel, sub)`, `.whereExists(rel, sub)`,
`.where(col, [op,] val)`, `.cmp(col, [op,] val)`, `.and`, `.or`, `.not`.

### 4.1 `where/server.ts` (all cross-table EXISTS over server relations)

| Predicate | Family | Shape | Cross-table? |
| --- | --- | --- | --- |
| `hasServerOwnerPermission` :4 | allow | EXISTS server WHERE creatorId=me OR userId=me | yes (server) |
| `hasServerAdminPermission` :13 | allow | EXISTS server → EXISTS userRoles(userId=me, canAdmin) | yes (userRole) |
| `hasServerChannelManagePermission` :21 | allow | EXISTS userRoles(me, canAdmin\|canEditChannel\|canEditServer) | yes |
| `hasServerAppManagePermission` :37 | allow | EXISTS userRoles(me, canAdmin\|canEditApp\|canEditServer) | yes |
| `hasServerModeratePermission` :53 | allow | EXISTS userRoles(me, canAdmin\|canModerate) | yes |
| `hasServerMemberPermission` :65 | allow | EXISTS server → EXISTS serverMembers(userId=me) | yes (serverMember) |
| `hasServerReadPermission` :73 | allow | EXISTS server WHERE private=false OR EXISTS serverMembers(me) | yes |

### 4.2 `where/channel.ts` (server membership + channel-role EXISTS, incl. junction)

- `hasChannelSelfReadPermission` :39 — `_.and(deleted != true,
  hasServerReadPermissionOnChannelSelf, _.or(private=false,
  EXISTS channelUserRoles(me), isServerAdmin))`. Uses the **junction**
  `channelUserRoles` (:43) and nested server EXISTS.
- `hasServerReadPermissionOnChannelSelf` :28 — EXISTS server WHERE (private=false
  OR EXISTS serverMembers(me)).
- `hasServerAdminPermissionOnChannelSelf` :20 — EXISTS server → EXISTS
  userRoles(me, canAdmin).
- `canReadChannelSelf` :60 — table-bound wrapper of the above for `can()` in
  mutations.
- `hasChannelReadPermission` :72 — server member AND (channel not private OR
  channel role OR server admin), all EXISTS.
- `hasChannelMessageWritePermission` :95, `hasChannelWritePermission` :105,
  `hasChannelEditRole` :87, `hasChannelRole` :65 — deny/allow write families,
  all cross-table EXISTS.

### 4.3 `where/message.ts` (channel read + solo-channel + NOT EXISTS)

- `hasMessageReadPermission` :4 — admin bypass; else `_.and(deleted != true,
  hasChannelReadPermission, soloCheck)` where `soloCheck = _.or(
  _.not(_.exists('channel', solo=true)), creatorId=me, type='bot')`
  (`:10-14`). **NOT + EXISTS** plus a transitive channel-read EXISTS chain.
- `hasRelatedMessageReadPermission` :19 — same logic wrapped in
  `EXISTS message(...)` for `messageReaction`.

### 4.4 `where/data.ts` (membership + row-local secret/permission gate)

- `hasDataReadPermission` :23 — `_.and(isServerMember (EXISTS serverMembers),
  permissions != 'secret', _.or(public, creator, owner, hasViewRole
  (EXISTS userRoles canViewData), admin))`. Mixes **cross-table EXISTS** with
  **row-local** column checks. The `!= 'secret'` is the guard that keeps
  secret rows (API keys, OAuth tokens) out of every client store.
- `hasDataWritePermission` :43, `hasDataViewRole` :5, `hasDataEditRole` :14 —
  role EXISTS families.

### 4.5 Row-local vs cross-table summary

- **Purely row-local** (no EXISTS): the column parts of `data` (secret/public/
  creator/owner), `message` solo creator/bot, most `*ByUserId`/`*.own`
  ownership checks in per-user tables (`channelNotificationSetting`,
  `seen`, `userState`, `notification` — own `userId`).
- **Cross-table EXISTS** (the reason query-aware sync is a Chat security
  requirement): every `server.*`, `channel.*` read/write predicate, message
  read, data membership, `userRolesByRole`, `appInstall` read,
  `channelActive` read — all gate on `serverMembers`, `userRoles`, or
  `channelUserRoles` (a junction) within the namespace.

## 5. Mutators (M4c: run inside the host transaction)

Every model namespace (`src/data/generated/models.ts`, 55 namespaces) exports
one `mutate` dispatcher built by `mutations(table, writePermission, {actions})`
(on-zero `mutations.ts:112-219`). CRUD (`insert`/`upsert`/`update`/`delete`)
is auto-added with a `can(writePermission, obj)` check — **after** the write
for insert/upsert, **before** for update/delete (`mutations.ts:178-194`).
Custom actions live alongside (e.g. message: `send`, `sendParsed`, `sendDraft`,
`save`, `insertOptimisticBotReply`, `insertBulk`, `deleteAll` —
`mutations/message.ts:77-297`).

Authoritative per-action arg validators: `src/data/generated/syncedMutations.ts`
(2842 lines, valibot). Non-CRUD namespaces with extra exported helpers:
`channel` (createBetaSharedChannel, getSharedChannelIds, channelExistsInSort,
addChannelToSort, addChannelToSortGroup), `app` (insertAppThread), `flow`
(insertFlowThread).

### 5.1 Transaction / context interface (the DO-local adapter contract)

Each action is `(ctx: MutatorContext, obj) => Promise<void>`. `MutatorContext`
(on-zero `types.ts:68-77`) — **identical to what soot's mutators receive**, so
M4c and M4a share this contract:

```ts
MutatorContext = {
  tx: Transaction            // ZeroTransaction<Schema>
  authData: AuthData | null  // normalized claims: { id, role, email, ... }
  environment: 'server' | 'client'
  server?: { actions: ServerActions; asyncTasks: Array<() => Promise<void>> }
  can: Can                   // (where, obj) => Promise<void>, throws on deny
}
```

Surface the DO-local SQLite adapter must provide (observed in
`mutations/message.ts`):

- `tx.mutate.<table>.{insert,upsert,update,delete}(row)` — e.g. `:79`,
  `:103`, `:112`, `:244`.
- `tx.run(query)` — read inside the tx, e.g. `:86` `tx.run(zql.message
  .where('id', ...).one())`.
- module-level `run(query)` (on-zero) for SSR reads — e.g. `:289`, `:307`.
- `ctx.can(permission, obj)` — throws → Zero rolls back the tx.
- `ctx.authData` — id/role for permission and creator resolution.
- `ctx.server.asyncTasks.push(fn)` — deferred **post-commit** effects.
- `ctx.server.actions.<...>` — consumer server-action callbacks
  (e.g. `server.actions.pg.deleteRow(...)` `:414`,
  `server.actions.server.refreshPrivateChatStatsForMessage(...)` `:608`).

### 5.2 Tables written beyond the target (derived-state writes in the same tx)

Mutators write more than their own table. `message.send` /`insert` writes
`message`, then `applyMessageDerivedState` (`:573-592`) updates channel latest
order + `recomputeThreadReplyCount` (thread), inserts `thread` on auto-thread
(`:428`), `attachment` (`:542`), and mention join rows `messageUserMention` /
`messageRoleMention` / `messageChannelMention` (`:377-398`). Any M4c transaction
adapter must let a mutator touch several tables atomically.

### 5.3 External side effects (deferred, post-commit only)

21 mutation files enqueue external effects, all via
`ctx.server.asyncTasks.push` or `server.actions.*`, gated by
`process.env.VITE_ENVIRONMENT === 'ssr'` so they run only server-side and only
after commit. Effect categories (source dirs `src/data/server-effects/{app,
appInstall,data,flow,meeting,message}`):

- notifications + push + email (`notification` table has `emailedAt`/
  `emailBatchId`/`emailClaimedAt`; `server-effects/message`),
- search indexing, AI/agent dispatch ("HUD"), mirror-to-shared-channel
  (`message.ts:464-483`),
- private-chat stats refresh (`message.ts:594-613`),
- typing-indicator clear (`message.ts:412-418`),
- flow/webhook execution (`server-effects/flow`), meeting effects.

This matches the plan's rule (invariant 17): external side effects run only
after mutation commit. The engine must collect `asyncTasks` and run them
post-commit, exactly as the on-zero server shim does today (see soot's
`handleMutationRequest`, `soot-migration-inventory.md` §B).

## 7. Minimal AST subset M4b must support for Chat (the gate)

This is the section that gates M4b compiler scope. Everything below is
**exercised by production Chat query traffic** (the `useQuery`-reached queries
in §3; devtools/sandbox excluded). Each item is a unit-test + real-client
harness case per the plan.

### 7.1 Required AST (compile all of these)

1. **Conditions / operators** on a column vs a bound value:
   `=`, `!=`, `<`, `<=`, `>`, `>=`, `IN`, `LIKE`, `ILIKE`, `IS` (NULL),
   `IS NOT` (NOT NULL). Values are string / number / boolean / null and, for
   `IN`, arrays of string.
2. **Boolean structure**: `AND`, `OR`, `NOT`, nested arbitrarily (§3.2, §4).
3. **Correlated EXISTS** subqueries over relationships, nested to **3 levels**
   (`channelMembers`: user→roles→channelPermissions) and combined with `NOT`
   (`where/message.ts` solo check) and `OR`.
4. **Junction (many-to-many) EXISTS and related**: expand a two-hop
   relationship through its junction table (`channelUserRoles`,
   `creator.roles`, `friends`, `userRoles`) into two joins. 12 junctions in
   the schema; one (`channelUserRoles`) is inside a security predicate.
5. **Related rows** (`related`) nested to **~4 levels**
   (`channelMessages` → messages → thread → appInstalls → app), including
   `.one()` relations, per-relation `where`, per-relation `orderBy`+`limit`.
6. **Ordering** with a stable **composite-PK tie-breaker** (9 tables have
   composite PKs, §1.2), including **multi-key orderBy** (`hotThreads`:
   `replyCount desc, updatedAt desc`) and **runtime-chosen sort columns**
   (`dataSearch`, `jobsSorted`).
7. **Limits**, including `limit(0)` (`topPrivateChats`) and `min(1000, n)`
   caps; `.one()` = LIMIT 1.
8. **Cursor pagination via comparison operators** (`where(col, </>, cursor)`
   + `orderBy` + `limit`) — **not** `.start()`.
9. **Parameter binding** with positional `?` and **no** SQL string
   interpolation; arrays bind to `IN (?, ?, …)`. Bind types: string, number,
   boolean, null, string[].

### 7.2 FLAGGED — used by Chat, NOT named in the plan's supported-AST list

The plan's "Supported AST" (final-plan §"Query execution") lists only:
equality/comparison, and/or/not, correlated EXISTS, related rows, ordering
with PK tie-breaker, limits, **start cursors**, parameter binding. Chat needs
more. **These must be added to M4b scope or the compiler will reject valid
production queries:**

1. **`ILIKE`** (11 sites, 5 files) — the biggest flag. SQLite has **no
   `ILIKE` keyword**. The consumer host emits Zero's `ILIKE` op; the compiler
   must map it to case-insensitive matching (`col LIKE ? COLLATE NOCASE` or
   `LOWER(col) LIKE LOWER(?)`). Postgres `ILIKE` is Unicode-case-insensitive;
   SQLite `LIKE`/`NOCASE` is ASCII-only by default — decide and test the
   collation, since search results diverge otherwise. Used by user/member
   search, message pinned-search, data key search
   (`server.ts:90`, `channel.ts:100`, `message.ts:70`, `data.ts:31`,
   `user.ts`).
2. **`LIKE`** (3 sites) — pattern match with caller-supplied pattern
   (`data.dataByKeyLike:181`, `user.testUsers`, `devtools.exploreTable`).
   Also not "equality/comparison".
3. **`IN`** (7 sites) — array membership. Treat as first-class (with
   **empty-array** semantics: `IN ()` must yield no rows, not a SQL error).
4. **`IS` / `IS NOT` NULL** — null tri-value handling. A plain `=`/`!=` does
   not match/aggregate NULL in SQL; the compiler must emit `IS NULL` /
   `IS NOT NULL` distinctly (`server.ts:143`, `channel.ts:211`,
   `message.ts:38`).
5. **Junction (many-to-many) relationships** — the plan says "related rows"
   and "correlated EXISTS" but does not mention that Chat's relationships
   include 12 two-hop junctions that expand to an intermediate table; one is
   load-bearing for **channel read security** (`channelUserRoles`). Must be in
   scope, not treated as a simple single-hop relation.
6. **Runtime-dynamic `orderBy` columns and filters** — `dataSearch`,
   `jobsSorted`, `exploreTable` pass a `sortColumn` string chosen at runtime.
   The AST column is validated by the query's arg validator but is not
   statically fixed; the compiler must accept any real column, not an
   enumerated allow-list per query.
7. **`.start()` is NOT used** — the plan lists "start cursors" as supported,
   but **zero Chat queries use `.start()`**. Chat cursors entirely via
   comparison operators (item 7.1.8). Supporting `.start()` is not required
   for Chat; supporting comparison-operator cursoring **is**. (Confirm Soot's
   need separately — soot-migration-inventory.md.)

### 7.3 Explicitly out of scope for Chat production

- The markdown **sandbox** query (`MarkdownSandboxQuery.tsx`) — arbitrary
  user-authored `zero.query.<table>` shape; devtools/markdown only. Reject or
  scope out; do not try to support arbitrary shapes.
- `devtools.*` queries (`exploreTable` dynamic table, unfiltered
  `allFriendships`, `waitlistAll`) — devtools console only, not in the e2e
  permission suite. Not required for the M4c end-to-end gate, but note
  `exploreTable` builds a query over a **runtime table name** — if kept, it is
  a `zql[table]` shape the compiler cannot statically bound.
- No aggregation (`count`/`sum`/`groupBy`/`having`/`distinct`) appears in any
  query — Chat precomputes counts into tables (`messageReactionStats`,
  `thread.replyCount`, `privateChatsStats`) via mutators. The compiler needs
  **no** aggregation support.

### 7.4 Bottom line for M4b

The Chat query surface fits the plan's recomputation design, but the compiler
scope must be widened past the plan's written AST list to add **ILIKE, LIKE,
IN (with empty-array), IS/IS NOT NULL, junction (two-hop) relationships, and
runtime sort columns**, and can **drop** `.start()` cursors in favor of
comparison-operator cursoring. The security-critical additions are `ILIKE`
(most-used) and junction expansion for `channelUserRoles` (channel read
permission). Composite PKs (9 tables) make the tie-breaker and cursor logic
multi-column, not single-`id`.
